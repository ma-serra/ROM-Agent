Antes de qualquer procedimento de instalação, o técnico deve executar diagnóstico
completo do ambiente macOS:

COMANDO 1: Versão do sistema operacional
sw_vers
Saída esperada: macOS Big Sur (11.x), Monterey (12.x), Ventura (13.x) ou
Sonoma (14.x). Versões anteriores a Big Sur (10.16) podem apresentar
incompatibilidades com Python 3.9+.

COMANDO 2: Verificar processador
uname -m
Saída esperada: arm64 (Apple Silicon M1/M2/M3/M4) ou x86_64 (Intel).
CRÍTICO: Versões Intel do Python diferem de Apple Silicon. Verificar
compatibilidade do interpretador.

COMANDO 3: Verificar shell padrão
echo $SHELL
Saída esperada: /bin/zsh (padrão em Catalina+) ou /bin/bash.
Obs: Diretivas de configuração (.bashrc, .zprofile) dependem do shell.

A.2. VERIFICAÇÃO DE DEPENDÊNCIAS CRÍTICAS DO SISTEMA

PASSO 1: Verificar Python instalado
python3 --version
Saída esperada: Python 3.9.x ou superior. Se ausente: instalar via
Homebrew (brew install python3) ou official installer (python.org).

PASSO 2: Verificar Pip
pip3 --version
Saída esperada: pip 21.0+ associado ao Python 3.9+.
Se ausente: python3 -m pip install --upgrade pip

PASSO 3: Verificar Git
git --version
Saída esperada: git 2.30.0+.
Se ausente: brew install git (requer Xcode Command Line Tools).

PASSO 4: Verificar Homebrew
brew --version
Saída esperada: Homebrew 3.x ou superior.
Se ausente: /bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

SITUAÇÃO: V2.0 admitia iniciar SCEAP manualmente com:
  python3 /path/to/sceap_v2.py

V5.0 NÃO suporta este método direto. Exige:
  python3 -m sceap.auto_update.cli check -r requirements.txt

Admin quer usar método antigo de V2.0.

QUESTÃO: Método de V2.0 continua válido em V5.0?

RESPOSTA (OVERRULING — V5.0 REVOGA EXPLICITAMENTE V2.0):

NÃO. Método de V2.0 é REVOGADO (overruling), pelos seguintes motivos:

1. MUDANÇA ARQUITETURAL: V5.0 reorganizou package structure
   /modulos/sceap_v2.py → /auto_update/cli.py (novo namespace)

2. CLI MODULAR: V5.0 usa interface CLI robusta (via Click library)
   Método direto (python3 sceap.py) é inseguro em V5.0

3. PRECEDENTE VINCULANTE: Documentação V5.0 (este prompt) é
   normativa (art. 927 CPC analogia) para formato oficial

SOLUÇÃO (OVERRULING SEGURO):

Admin que precisa de compatibilidade máxima pode criar wrapper:

# /custom_modules/sceap_v2_compat.py
#!/usr/bin/env python3
"""Wrapper de compatibilidade V2.0 → V5.0 (DEPRECATED)"""

import subprocess
import sys

def main():
    # Redireciona comando antigo para novo
    cmd = ["python3", "-m", "sceap.auto_update.cli", "check", 
           "-r", "requirements.txt"]
    result = subprocess.run(cmd)
    sys.exit(result.returncode)

if __name__ == "__main__":
    main()

# Uso: python3 sceap_v2_compat.py
# Internamente: chama V5.0 cli corretamente

CONSEQUÊNCIA:

1. Método V2.0 (python3 sceap.py) é REVOGADO
2. Método V5.0 é OBRIGATÓRIO (python3 -m sceap.auto_update.cli)
3. Compatibilidade é possível via wrapper (cria ilusão de V2.0)
4. Wrapper deve ter aviso: "DEPRECATED: use V5.0 CLI directly"

SEÇÃO E: MATRIZ DE DECISÃO — DISTINGUISHING VS OVERRULING

────────────────────────────────────────────────────────────────────────────────

Quando encontrar conflito V2.0 vs V5.0, usar matriz:

┌──────────────────┬─────────────────┬──────────────────┬──────────────────┐
│ TIPO DE CONFLITO │ PODE COEXISTIR? │ DECISÃO          │ AÇÃO             │
├──────────────────┼─────────────────┼──────────────────┼──────────────────┤
│ Configuração     │ NÃO             │ DISTINGUISHING   │ Migrar/descartar │
│ Dependência      │ SIM (venv)      │ DISTINGUISHING   │ Isolação         │
│ Estrutura dir    │ PARCIAL         │ DISTINGUISHING   │ Preservar dados  │
│ Formato código   │ NÃO             │ OVERRULING       │ Reescrever       │
│ API/Interface    │ NÃO             │ OVERRULING       │ Wrapper (temp)   │
│ Dados (output)   │ SIM             │ COMPATIBILIDADE  │ Converter        │
│ Segurança        │ NÃO             │ OVERRULING       │ Atualizar        │
└──────────────────┴─────────────────┴──────────────────┴──────────────────┘

LÓGICA:
- Se pode coexistir: DISTINGUISHING (preservar, isolar)
- Se não pode coexistir: OVERRULING (revoga V2.0, força V5.0)
- Se é dados: compatibilidade (converter se possível)
- Se é segurança: sempre OVERRULING (V5.0 obrigatório)

SEÇÃO E: DISSÍDIO 5 — CERTIFICAÇÃO E ASSINATURA

────────────────────────────────────────────────────────────────────────────────

TESE A (Sem assinatura): Relatórios são entregues como plain text/PDF
  Qualquer admin pode criar relatório fictício

TESE B (Com assinatura digital): Relatórios são assinados com chave

  Apenas holder da chave privada pode assinar relatórios válidos
  Receptor pode verificar integridade com chave pública

RESOLUÇÃO (STJ-ANALÓGICO):

TESE B (Com assinatura) é OBRIGATÓRIA para instalações de PRODUÇÃO.
Razão: Conformidade regulatória, prova de autenticidade em potencial
disputa.

IMPLEMENTAÇÃO:

# 1. Gerar par de chaves (uma vez)
openssl genrsa -out ~/.sceap/private.key 2048
openssl rsa -in ~/.sceap/private.key -pubout -out ~/.sceap/public.key

# 2. Assinar relatório
python3 -m sceap.auto_update.cli report -r requirements.txt \
  -o ~/relatorio.md \
  --sign-with ~/.sceap/private.key

Saída: relatorio.md + relatorio.md.sig (assinatura)

# 3. Verificar assinatura (receptor)
openssl dgst -sha256 -verify ~/.sceap/public.key \
  -signature relatorio.md.sig relatorio.md

Resultado: "Verified OK"

CONSEQUÊNCIA:

Para PRODUÇÃO: gerar chaves criptográficas e assinar relatórios.
Para DESENVOLVIMENTO: assinatura é opcional.
